# 랜덤채팅 정책 결정 체크리스트 (2026-04-10)

## 이미 반영된 기준
- 인접 연령 허용 범위: 30대 ↔ 40대만 허용
- 차단 해제 로그: 차단 해제 시 관리자 로그 무조건 기록
- 영구정지 실행 기준: 영구정지 시 기존 채팅 스레드는 유지
- 메시지 삭제 복구: 관리자만 원문 복구 가능
- 운영 DB 방향: Railway PostgreSQL 기준

## 아직 최종 결심이 필요한 항목

### 1) 거리 매칭 점수식
- 결정 포인트: `band_bonus`(구간형 가점)로 갈지, `linear_decay`(거리 비례 감점)로 갈지 확정 필요
- 권장안: 초기 운영은 `band_bonus`
- 이유: 운영자가 설명하기 쉽고, 민원 대응 시 기준이 단순하며, 과도한 미세 거리 차별을 줄일 수 있음
- 권장 구간안: 0-10km +30 / 10-30km +20 / 30-70km +10 / 70km 초과 +0
- 결심 기준: 민원 최소화와 운영 단순성이 우선이면 구간형, 정밀 매칭 만족도가 우선이면 선형식

### 2) 차단 해제 권한 범위
- 결정 포인트: 사용자 자가 해제 허용 유지 여부
- 권장안: 일반 사용자는 자기 차단 해제 가능, 관리자 해제는 예외조치로만 운영
- 이유: 일반 분쟁은 자가 복구가 빨라야 하고, 관리자는 감사로그 중심으로 개입하는 편이 효율적
- 결심 기준: 악성 재접촉 위험이 크면 관리자 전용 해제로 축소

### 3) 영구정지 후 기존 채팅 노출 범위
- 결정 포인트: 기존 스레드를 완전 열람 유지할지, 읽기전용 잠금으로 둘지
- 권장안: 기존 스레드는 읽기전용 유지, 신규 메시지 전송만 차단
- 이유: 분쟁/증거 보존과 사용자 체감 모두 균형적
- 결심 기준: 증거보존이 최우선이면 읽기전용 유지, 재노출 리스크가 크면 관리자 전용 보관소 이동

### 4) 메시지 삭제 정책 세부
- 결정 포인트: 사용자 삭제를 화면 마스킹으로 둘지, 실제 삭제 요청 큐로 보낼지
- 권장안: 사용자 삭제는 화면상 `삭제된 메시지` 마스킹, DB 원문은 관리자 전용 보관
- 이유: 신고/분쟁/수사 협조 대응 가능
- 결심 기준: 법적 보존이 중요하면 마스킹+보관, 프라이버시 강조가 더 크면 보존기간 단축 규칙 추가

### 5) 신고 누적에 따른 자동정지 임계값
- 결정 포인트: 자동 임시정지 건수와 기간
- 권장안: 최근 30일 고유 신고 3건 이상 = 검토대기, 5건 이상 = 72시간 임시정지
- 이유: 오탐을 줄이면서 반복 악성 사용자를 빠르게 격리 가능
- 결심 기준: 초기 유저풀이 적으면 임계값을 더 높게, 커뮤니티 변동성이 크면 더 낮게

### 6) 지역 공개 수준
- 결정 포인트: 매칭 결과에 시/도까지만 보여줄지, 시/군/구 일부까지 보여줄지
- 권장안: 기본은 시/도 또는 광역권만 공개
- 이유: 프라이버시와 안전 측면에서 보수적 운영이 유리
- 결심 기준: 실제 매칭 만족도보다 신원 노출 리스크를 더 낮추고 싶으면 광역권까지만 공개

### 7) WebSocket 운영 구조
- 결정 포인트: Railway 단독 운영인지, Redis 브로커 추가인지
- 권장안: 초기 MVP는 Railway 단독, 동접/다중 인스턴스 확대 시 Redis 추가
- 이유: 지금 단계에서는 운영 단순성이 우선
- 결심 기준: 동시 접속 증가와 메시지 유실 우려가 커지면 Redis 전환

## 최종 권장 결론
- 거리 점수식은 `band_bonus`로 먼저 운영
- 영구정지 사용자는 기존 채팅 읽기전용 유지
- 메시지 삭제는 사용자 마스킹 + 관리자 복구 전용
- PostgreSQL 단일 운영을 기준으로 마이그레이션 고정
- 신고 임계값은 30일 3건 검토 / 5건 72시간 정지로 시작
