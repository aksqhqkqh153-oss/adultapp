# 랜덤채팅 정책 결정 반영본 (2026-04-10)

## 이번에 확정한 운영 기준
- 거리 점수식: `band_bonus` 구간형 가점으로 운영
- 인접 연령 허용: `30대 ↔ 40대`만 허용
- 차단 해제 권한: 일반 사용자 자가 해제 허용
- 차단 해제 로그: 차단 해제 시 관리자 로그 무조건 기록
- 영구정지 후 기존 채팅: 읽기전용 유지 + 관리자 전용 보관소 동시 저장
- 메시지 삭제: 사용자 화면에서는 `메세지가 삭제되었습니다` 마스킹 표기 + 관리자 전용 보관소 동시 저장
- 지역 공개 수준: `시`까지만 공개
- WebSocket 운영: 초기에는 Railway 단독 운영, 트래픽 증가 시 Redis 검토
- 운영 DB 방향: Railway PostgreSQL 기준으로 통일

## 현재 스타터 기준 신고 자동정지 임계값
- 현재 코드 기준 자동정지 정책: `5:3d,10:7d,20:30d,21:admin_review`
- 의미
  - 신고 누적 5건 이상: 3일 임시정지
  - 신고 누적 10건 이상: 7일 임시정지
  - 신고 누적 20건 이상: 30일 임시정지
  - 신고 누적 21건 이상: 관리자 검토 상태 전환
- 현재 코드 기준 자동정지 임계 숫자 필드: `auto_suspend_threshold = 5`

## 운영 권장 판단
- 초기 운영은 그대로 `5건 -> 3일`로 시작하되, **고유 신고자 수 기준**을 같이 적용하는 편이 안전합니다.
- 권장 해석
  - 동일인이 반복 신고한 건은 1명으로 묶기
  - 최근 30일 기준으로 계산
  - 즉시 자동정지 전 `욕설/음란물전송/개인정보요구` 같은 고위험 사유는 가중치 부여
- 실무 권장안
  - 최근 30일 `고유 신고자 3명` 이상: 관리자 검토 큐
  - 최근 30일 `고유 신고자 5명` 이상: 24~72시간 임시정지
  - 동일 사유 반복 + 증거 첨부 신고 다수: 즉시 수동 검토

## 추가로 결심이 필요한 항목

### 1) 신고 집계 기준
- 결정할 것: 단순 신고 건수 기준인지, `고유 신고자 수` 기준인지
- 권장: `최근 30일 고유 신고자 수 + 신고 사유 가중치`
- 이유: 동일인의 악의적 반복신고 오탐을 줄일 수 있음

### 2) 신고 사유별 가중치
- 결정할 것: 모든 신고 사유를 동일 점수로 볼지
- 권장: `욕설=1`, `스팸=1`, `개인정보요구=2`, `음란물전송=3`, `불법권유=3` 같은 가중치 테이블 운영
- 이유: 위험도가 다른 신고를 같은 기준으로 다루면 조치 정확도가 떨어짐

### 3) 읽기전용 스레드 상세 정책
- 결정할 것: 영구정지된 상대의 기존 채팅을 상대방에게 어디까지 보여줄지
- 권장: 기존 내역 열람 가능, 신규 전송/파일첨부/프로필 진입 차단
- 이유: 분쟁 증거 보존과 사용자 안내를 동시에 만족시킴

### 4) 관리자 보관소 접근 권한
- 결정할 것: 누구까지 보관소 열람 허용할지
- 권장: `관리자(grade 1)` 기본, 필요 시 `감사 전용 서브권한`만 추가
- 이유: 삭제 메시지 원문과 제재 기록은 최소권한 원칙이 필요함

### 5) 보관 기간
- 결정할 것: 삭제 메시지/정지 스레드/신고 로그 보관 기간
- 권장: 운영 로그 1년, 삭제 메시지 원문 90~180일, 영구정지 증거 1년 이상
- 이유: 분쟁 대응과 저장비용 균형이 필요함

### 6) 지역 노출 방식
- 결정할 것: `서울특별시`처럼 전체 명칭을 그대로 보여줄지, `서울`처럼 축약할지
- 권장: 사용자 표시값은 `서울`, `인천`, `안양`, `김포`, `고양`처럼 시 단위 표준화
- 이유: UI 가독성과 개인정보 최소노출에 유리함

### 7) WebSocket 확장 전환 시점
- 결정할 것: 어느 시점부터 Redis 브로커를 붙일지
- 권장: 동접 급증, 다중 인스턴스, 메시지 유실 징후 중 하나라도 발생하면 Redis 전환
- 이유: 초기에는 Railway 단독이 단순하지만, 확장 시 세션 동기화가 필요함

### 8) PostgreSQL 운영 규칙
- 결정할 것: SQLite 잔존 허용 범위
- 권장: 로컬 개발/테스트만 SQLite, 운영은 PostgreSQL only
- 이유: 랜덤채팅/신고/감사로그는 운영 DB 일관성이 중요함
